# XR Sandbox Build Plan

## Objective
Build the smallest honest XR surrogate that proves whether `speedrun` can evolve from an ambient ident into a navigable signal experience.

## Build Principle
Preserve the current sense of pressure and momentum, but only allow motion that teaches state change or user consequence.

## Target Environment
- Seated or standing headset demo
- Session length: `2 to 4 minutes`
- Safe reach model: no repeated elevated-arm gestures
- Interaction comfort: horizon-stable environment, limited camera travel, oversized targets

## Interaction Model
- Primary input: gaze dwell over lane markers or state nodes
- Secondary input: controller trigger confirm
- Risky action confirmation: two-step confirm for any scene transition or system commit
- Fallback mode: passive autoplay with narrated chapter cards

## Scene Plan
### Phase 1: Honest Mirror
- Recreate the current ident in depth-lite form.
- Keep the logo central.
- Move background plate, wake trails, and network layer into shallow depth bands.
- Add a visible disclosure label stating that only the motion surface is currently validated.

### Phase 2: Named Lanes
- Replace decorative nodes with `3 to 4` named lanes or chapters.
- Give each lane a stable spatial anchor and distinct pulse behavior.
- Relocate the logo into a smaller state badge during selection moments.

### Phase 3: Consequence
- Add one success path and one failure path.
- Make wake behavior, pulse speed, and color shift correspond to state outcome.
- Record latency, target hit rate, and comfort notes from internal review.

## Technical Slice
- Engine options:
  - Unity for fastest general XR sandboxing
  - WebXR for browser-adjacent continuity
  - TouchDesigner only if the target is presentation-first rather than headset-first
- Asset carryover:
  - logo texture
  - backplate stills as provisional skybox or panel textures
  - wake, streak, and pulse behavior as shader or particle references

## Validation Gates
1. Users can identify what is interactive within `5s`.
2. Users can complete one selection without arm fatigue or ambiguity.
3. The scene sustains `72fps` on target hardware.
4. At least one visual layer maps to real system state instead of pure atmosphere.
5. Every speculative behavior is labeled in the build notes.

## Exit Criteria
- If the sandbox still looks impressive but teaches nothing, stop and keep the original project as a hero ident.
- If named lanes, consequence states, and comfort targets hold together, continue into a more complete prototype.
